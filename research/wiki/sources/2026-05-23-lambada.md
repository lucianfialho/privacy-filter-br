# Do Not Have Enough Data? Deep Learning to the Rescue! (LAMBADA)

Anaby-Tavor et al., IBM Research AI — arXiv [1911.03118](https://arxiv.org/abs/1911.03118)

## Abstract (paraphrased)

For text classification with very few labeled examples (5-100 per class), how do you get a strong classifier? Anaby-Tavor et al. propose **LAMBADA** (Language-Model-Based Data Augmentation): fine-tune GPT-2 on the small labeled set, generate 10× the desired synthetic examples per class, then **filter** the synthesized examples using a baseline classifier trained on the original data — keep only those the classifier confidently labels with the matching class. Trains a final classifier on `(original ∪ filtered synthetic)`. Beats EDA, CVAE, and CBERT on ATIS, TREC, and WVA across SVM/LSTM/BERT classifiers. Even outperforms semi-supervised approaches that use *real* unlabeled data.

## The pipeline (Algorithm 1)

```
Input:  Small training set D_train, classifier algorithm A,
        language model G (GPT-2), targets N_1..N_q per class

Step 1: Train baseline classifier h = A(D_train)
Step 2: Fine-tune G on D_train with format
        "label_1 [SEP] sentence_1 [EOS] label_2 [SEP] sentence_2 [EOS] ..."
        → G_tuned

Step 3: For each class y, sample 10*N_y sentences from G_tuned
        by conditioning on "y [SEP]" and generating until [EOS]
        → D* (raw synthesized pool)

Step 4: Filter D* using h:
        - Verify h(x) == y (predicted label matches generation label)
        - Rank by h's confidence score
        - Keep top N_y per class
        → D_synthesized

Output: Train final classifier on D_train ∪ D_synthesized
```

## Key findings

- **Filter step is essential.** Generate 10× more than needed, keep only the 10% that the baseline agrees with. Without filtering, generated sentences are too noisy.
- **Works across classifier types.** SVM, LSTM, BERT all benefit. Method is classifier-agnostic.
- **Improvement is largest when data is smallest.** ATIS @ 5 samples/class: BERT goes from 53.3 → 75.7 accuracy (+22.4 pts). At 100 samples/class the gain is much smaller (LSTM/SVM see no improvement).
- **Beats simpler augmentation methods.** EDA (synonym swap), CVAE (autoencoder), CBERT (BERT mask-predict). LAMBADA statistically superior (McNemar p<0.01).
- **Beats semi-supervised with real unlabeled data.** Curious finding: LAMBADA-synthesized data beats simple weak-labeling of real unlabeled data. Suggests the LM injects useful inductive bias not present in the unlabeled distribution alone.
- **Handles imbalanced classes naturally.** Generate exactly N_y per class, regardless of original distribution. Biggest gains came on ATIS (imbalanced) — +58% relative improvement for BERT.

## How this relates to our work

This is the **methodological closest** paper to our pipeline so far. Our dataset construction is structurally similar:

| | LAMBADA | our v3 pipeline |
| --- | --- | --- |
| LM used | GPT-2 (fine-tuned on small task data) | GPT-5-nano (NOT fine-tuned, used only for rewriting) |
| Conditioning | Class label `[SEP]` → sentence | Template + PII slots → sentence |
| Labeling | Generated jointly with text | Deterministic string-match against inserted PII |
| Filtering | Baseline classifier with confidence rank | **NONE — we keep everything** |
| Target regime | 5-100 examples per class | 2500+ examples per class |

**The biggest gap in our pipeline is the missing filter step.** We generate 54k examples and keep all of them, including:
- Examples where gpt-5-nano subtly drops or alters the inserted PII
- Examples where the rewrite shifts the entity boundary
- Examples that are stylistically homogeneous (LM mode collapse)

LAMBADA's recipe: train an interim model, use it to filter, retrain. This is **self-distillation via confidence filtering**. We don't do it.

## What we can actually borrow

1. **Classifier-confidence filtering on the existing 54k.** Train v3 on 80% of data, score the remaining 20% with v3. Drop any example where (a) string-match label disagrees with v3's predicted span, OR (b) v3's confidence is below threshold τ. Retrain on the filtered set. Repeat once or twice.

2. **Per-category generation balance.** Don't let templates determine the per-category ratio. Explicitly target N examples per category. Our `private_phone` has lowest F1 in v3 — likely under-represented. Same applies to long-tail categories (`private_voucher`, `private_mac_address`).

3. **Drift control via diversity metric.** Paper flags repeated synthesis can cause "biased synthesized samples gain domination". For us: track per-template diversity (unique surnames, unique street names, unique entity values) across generations. If diversity drops, stop or inject fresh values.

4. **Filter the LM output, not just the labels.** Our pipeline trusts gpt-5-nano's rewrites. We could add a check: does the rewritten sentence still contain all inserted PII spans? If gpt-5-nano dropped or altered a CPF in rewriting, throw that example out. Cheap to implement, would directly catch a known failure mode.

## Concerns / critique

- **LAMBADA targets small-data regime.** 5-100 examples/class. We're at ~2500/class. The paper itself notes gains shrink past 100 samples. **The "more synthetic data" prescription doesn't help us — we have plenty.** But the **filtering insight does transfer.**
- **Classification, not NER.** LAMBADA generates whole labeled sentences. For NER you also need consistent span annotations. The generate+filter pattern works for sequences, but the filter has to operate on (span, label) pairs not (sentence, label).
- **Paper is 2019, uses GPT-2.** Modern frontier LLMs (GPT-4, Claude, Gemini) produce much higher-quality synthetic text. The filter step matters less now because generations are cleaner — but the diversity/drift concerns still apply.
- **No real-world domain shift evaluation.** All experiments are train/test from the same dataset. Doesn't address our [[../questions/synthetic-data-quality]] H1 (synthetic-to-real gap). LAMBADA could synthesize a perfect held-out and still fail catastrophically on real-world OCR'd documents.
- **GPT-2 fine-tuning on 5 samples is risky.** They report it works, but fine-tuning a 1.5B model on 5 sentences is highly unstable. They don't report variance across runs.

## Open questions raised for us

1. **Would classifier-confidence filtering improve our v3 holdout F1?** Cheap experiment: train v3.1 on 80% of v3 data, drop low-confidence training examples, retrain on filtered set. Measure F1 on the original holdout.
2. **Does our gpt-5-nano rewriting drop PII?** Quick audit: for each of the 54k examples, check if every inserted PII span still appears in the rewritten output. Estimate: 1-5% drop rate would be plausible.
3. **Per-category re-balancing.** What's the actual per-category count in our dataset? Are the bottom-F1 categories under-represented? Need to check.
4. **Self-distillation iteration count.** LAMBADA does one filter pass. Would N iterations of {train, filter, retrain} converge or diverge? Untested.

## Direct contrast with DeID-GPT

[[../sources/2026-05-23-deid-gpt]] (DeID-GPT) and LAMBADA both use LLMs for PII/data tasks, but they're using LLMs for opposite roles:

- **DeID-GPT:** LLM at INFERENCE time. Prompt → redacted output. No training set augmentation.
- **LAMBADA:** LLM at TRAINING time. Generate synthetic examples → fine-tune a smaller classifier. LLM not in the inference loop.

For our project the LAMBADA pattern is more applicable: we want a small fast local model at inference, and we use the LLM only during dataset construction.
