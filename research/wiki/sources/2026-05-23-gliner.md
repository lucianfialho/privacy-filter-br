# GLiNER: Generalist Model for Named Entity Recognition using Bidirectional Transformer

Zaratiana et al., LIPN/CNRS — arXiv [2311.08526](https://arxiv.org/abs/2311.08526) — Code: [github.com/urchade/GLiNER](https://github.com/urchade/GLiNER)

## Abstract (paraphrased)

Traditional NER models are tied to a fixed taxonomy. LLMs (ChatGPT, UniNER) handle arbitrary entities via prompting but are slow and expensive. GLiNER bridges the gap: a **small bidirectional encoder** (DeBERTa-v3 base) that treats NER as **matching entity-type embeddings with span embeddings in latent space**. Zero-shot, GLiNER-L (300M) outperforms ChatGPT and UniNER-13B on standard NER benchmarks.

## Key findings

- **Architecture beats scale.** GLiNER-S (50M) outperforms ChatGPT and Vicuna-13B on zero-shot OOD NER (52.7 vs 47.5 vs 17.5 avg F1).
- **Span-based, not token-based.** Avoids BIOES fragmentation. All spans up to K=12 tokens are scored in parallel, complexity O(n) with span length cap.
- **Entity-type as prompt.** Input format: `[ENT] type1 [ENT] type2 [SEP] sentence`. The `[ENT]` token gets its own embedding learned during training. Zero-shot generalization comes from semantic similarity between type names and entity spans.
- **DeBERTa-v3 dominates as backbone.** BERT/RoBERTa lag behind, but even BERT-base achieves 49 F1 on OOD (still > ChatGPT).
- **Multilingual transfer works.** Trained only on English Pile-NER, GLiNER-Multi (mdeBERTa-v3-base) beats ChatGPT in 8/10 languages, including some non-Latin-script ones. NOT evaluated on PT-BR directly, but Spanish and Italian performance is strong.
- **Negative entity sampling matters.** 50% negative-entity ratio balances precision/recall. 0% (Pile-NER default) yields false-positive-heavy models.

## Training details (replicable)

- **Data:** Pile-NER (Zhou et al, 2023) — 44k passages, 240k spans, 13k distinct types, all from English Pile + ChatGPT-extracted entities.
- **Backbone:** DeBERTa-v3 base/large; multilingual variant uses mdeBERTa-v3-base.
- **Hyperparams:** AdamW, lr 1e-5 (backbone) / 5e-5 (FFN heads), dropout 0.4, 30k steps, 10% warmup, cosine decay. Max 25 entity types per sentence during training. K=12 max span length.
- **Loss:** Binary cross-entropy on (span × type) pairs.
- **Hardware:** GLiNER-L = 5h on a single A100.

## How this relates to our work

GLiNER addresses **three concrete gaps** in our v3 model:

1. **[[bioes-vs-span-based]]** — GLiNER's span-based architecture would eliminate the sub-token fragmentation of CPF/CNPJ/cartão that we patch with `merge_consecutive` in `br-pii-guardrail`. We'd get coherent spans for free.

2. **Adding new PII categories without retraining.** Today we'd need to regenerate 5-10k examples per new category and retrain v3. With a GLiNER-style model, we add the category name to the prompt and it works zero-shot (modulo quality of generalization). Examples: `private_pix_key`, `private_bank_routing`, `private_mac_address`.

3. **[[person-detection-failure-modes]]** — GLiNER's parallel matching could improve recall on compound names because spans are scored independently, not token-by-token. But this needs empirical validation on BR Portuguese specifically.

## Concerns / open questions for us

- **PT-BR is in the multilingual variant but not benchmarked.** Pile-NER is English-only. mdeBERTa-v3-base does cover PT, but the paper doesn't include PT in Table 3. Need to test ourselves.
- **K=12 span length cap.** BR addresses can easily exceed 12 tokens ("Rua das Flores Amarelas, Bairro Nossa Senhora de Fátima..."). Cap likely problematic for `private_address`.
- **Negative entity sampling at training time.** Means we need to maintain a vocabulary of "other PII types" during training, not just the 22 we care about. Or maybe the BR PII categories work as their own negative space.
- **Zero-shot ≠ free.** They train on Pile-NER (English) and transfer. For our case, we'd still need to fine-tune on BR PII to get checksum-validated CPF/CNPJ recognized correctly.
- **Decode is greedy + threshold 0.5.** Need to validate the threshold for our recall-priority guardrail use case.

## Comparison with our v3

| | v3 (BERTimbau + BIOES) | GLiNER (DeBERTa-v3 + span) |
| --- | --- | --- |
| Params | 110M | 50M-300M |
| Backbone | PT-BR pretrained | English/multilingual |
| Decoding | BIOES + simple aggregation | Span enumeration + greedy |
| New category support | Retrain | Add to prompt (zero-shot) |
| Sub-token fragmentation | Yes (patched in lib) | No (by design) |
| BR-specific PII (CPF checksum) | Yes (in lib regex layer) | Out of scope, needs lib too |
| F1 on our synthetic holdout | 0.9934 | Unknown, needs eval |

## Open thread → potential v4 experiment

**Could v4 = GLiNER fine-tuned on our 54k synthetic dataset?**

Plan would be:
1. Take GLiNER-Multi pretrained checkpoint
2. Fine-tune on `dataset_br_v3.jsonl` with our 22 entity types
3. Evaluate on `dataset_br_v3_holdout.jsonl`
4. Compare F1 + inference latency vs v3 (BERTimbau + BIOES)

If F1 is comparable AND it solves the fragmentation/extensibility issues, switch v4 to GLiNER.

Cost estimate: ~5h on A100 + a few hours wiring the data loader. Not free, but bounded.
