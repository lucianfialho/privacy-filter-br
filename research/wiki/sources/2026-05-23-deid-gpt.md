# DeID-GPT: Zero-shot Medical Text De-Identification by GPT-4

Liu et al. (UGA + Harvard MGH + Mayo Clinic + others) — arXiv [2303.11032](https://arxiv.org/abs/2303.11032) — Code: [github.com/yhydhx/ChatGPT-API](https://github.com/yhydhx/ChatGPT-API)

## Abstract (paraphrased)

Clinical notes contain PII protected by HIPAA. Rule-based and learning-based de-identification systems either lack generalizability or need expensive fine-tuning per scenario. The paper proposes **DeID-GPT**: feed clinical notes plus an explicit HIPAA-aware prompt to GPT-4 and let it redact identifiers in zero-shot. On the 2014 i2b2/UTHealth dataset, GPT-4 with explicit prompts reaches **0.99 entity-level accuracy**, beating fine-tuned ClinicalBERT (0.974), RoBERTa (0.947), BERT (0.798), Llama-2-7b (0.612), and Falcon-7b (0.605).

## Key findings

- **Prompt design is the dominant lever.** With an *implicit* prompt ("Please anonymize the following clinical note") ChatGPT scores 0.686 and GPT-4 scores 0.908. With an *explicit* prompt that names HIPAA categories with examples, ChatGPT jumps to 0.929 and GPT-4 to 0.99. That's +24 points on ChatGPT from prompt engineering alone.
- **Three-part prompt template** ([[../concepts/prompt-engineering-for-pii]]):
  1. Task statement ("redact identifiers per HIPAA")
  2. Command ("Replace all the following information with `[redacted]`")
  3. Specific rules (each HIPAA category, with a concrete example for each)
- **Small open-source LLMs fail.** Llama-1/2-7b, Falcon-7b ≈ 0.60-0.61 (essentially random). mT0 reaches 0.824, Flan-T5-base 0.737. Failure mode: "fundamental misunderstanding of the task" — repeating the prompt, generating nonsense.
- **Fine-tuned encoders beat instruction-tuned small LLMs.** ClinicalBERT 0.974 > Llama-2-7b 0.612. For domain-specific structured tasks like PII, small fine-tuned encoders > large general LLMs.
- **GPT-4 > GPT-3.5 (ChatGPT) even on bad prompts.** GPT-4 with implicit prompt (0.908) beats ChatGPT with explicit prompt (0.929)? Actually no — they're close, ChatGPT-explicit beats GPT-4-implicit on i2b2. The takeaway is GPT-4 is more *robust to prompt quality*, not strictly better at every prompt-quality level.

## How this relates to our work

This is the **paper closest to our problem statement**, but with major architectural differences:

| | DeID-GPT | privacy-filter-br |
| --- | --- | --- |
| Approach | Cloud LLM (GPT-4) + prompt | On-device small encoder (BERTimbau 110M) + fine-tune |
| Categories | HIPAA 18 | LGPD-aligned 22 |
| Cost per inference | ~$0.01-0.10 (GPT-4 API) | ~0 (local) |
| Latency | ~2-5s (API roundtrip) | <100ms (target) |
| Data residency | Sent to OpenAI | Local |
| HIPAA/LGPD audit | Not possible (closed model) | Possible (open weights) |
| Accuracy (their metric) | 0.99 | (different dataset, ~0.99 holdout — see below) |

The paper itself flags the data-residency problem in §VI: "patient data cannot be stored nor transmitted to a non-authorized external party". Our entire architecture choice (small local model + reversible AES-GCM tokenization) is the answer to that problem.

## What we can actually use from this

1. **Prompt template for an LLM fallback in `br-pii-guardrail`.** Currently we have 3 layers (regex, schema-aware, NER fallback). A fourth optional layer could be an LLM call for ambiguous cases — IF the user opts in to data being sent to a third party. The 3-part prompt structure is directly portable.

2. **Validation that synthetic surrogate data works.** i2b2/UTHealth uses real clinical notes with PII replaced by **realistic surrogates** (e.g., "James McCarthy" → "Joshua Howard"). That's exactly our 4devs + Jinja approach. The dataset is widely accepted as a de-id benchmark despite being synthetic.

3. **Strong evidence that small fine-tuned encoders beat small instruction-tuned LLMs.** ClinicalBERT 0.974 vs Llama-2-7b 0.612 on the same task. Validates our v3 architecture choice over a "throw a Llama at it" alternative.

4. **Cost calibration.** Paper cites $50/hr human annotation, $250k per 100M-word corpus. Our entire pipeline cost <$200 in API/compute. The economic case for synthetic-data + small-model is overwhelming.

## Concerns / critique

- **Metric choice is dubious.** They use entity-level accuracy = (TP+TN)/(TP+TN+FP+FN). For NER this can be inflated by abundant true negatives (most tokens are not PII). Standard practice is precision/recall/F1 per entity type. Their 0.99 number is not directly comparable to NER F1 scores.
- **N is small for GPT-4.** Only 50 manual samples (because no GPT-4 API at time of writing). Generalization claims rest on a small sample.
- **Single dataset.** i2b2 only. No external validity test. No cross-domain transfer.
- **Hype-heavy.** Half the paper is about ChatGPT applications in unrelated domains (social media, search engines, coding). Should be read as a "GPT-4 works for de-id, here's the prompt" demo, not a rigorous comparison.
- **No per-category error breakdown.** Which HIPAA categories did GPT-4 miss? Unknown. For us this matters — BR PII has CPF/CNPJ/CEP that don't exist in HIPAA, and we'd want to know if GPT-4 handles them.

## Open questions raised for us

1. **Would GPT-4 with our 22-category LGPD prompt score similarly on Portuguese?** The paper doesn't test PT. Worth a quick experiment: run GPT-4 with our prompt template on `dataset_br_v3_holdout.jsonl` and see if it beats v3.
2. **Can we extract distilled knowledge from GPT-4?** Use GPT-4 as a labeler on real BR documents (with appropriate redaction) and train a smaller model on those labels. Same as [[../questions/synthetic-data-quality]] but with GPT-4 as the labeler.
3. **What's the right error-analysis taxonomy?** Paper splits errors into "recognition limitation" (GPT-4) vs "task misunderstanding" (Llama). For our model the taxonomy is different — we'd split into "fragmentation", "type confusion", "missed entity", "false positive on common word". Worth formalizing for v4.

## HIPAA vs LGPD identifier mapping

Reference for our 22 categories vs HIPAA's 18:

| HIPAA category | Our LGPD-aligned category |
| --- | --- |
| 1. Names | `private_person` |
| 2. Geo < state | `private_address` |
| 3. Dates (not year) | `private_date_of_birth`, `private_date` |
| 4. Phone | `private_phone` |
| 5. Fax | (n/a in BR) |
| 6. Email | `private_email` |
| 7. SSN | `private_cpf` |
| 8. Medical record # | `private_medical_id` |
| 9. Health plan # | (could add) |
| 10. Account # | `private_bank_account` |
| 11. Certificate # | `private_passport`, `private_voter_id` |
| 12. Vehicle serial | `private_license_plate`, `private_vin` |
| 13. Device serial | `private_device_id` |
| 14. URLs | `private_url` |
| 15. IPs | `private_ip` |
| 16. Biometric | (n/a in our scope) |
| 17. Face photos | (n/a, text-only) |
| 18. Any unique ID | `private_rg`, `private_cnpj`, others |

We're not strictly HIPAA-compliant (BR is LGPD), but the overlap is high enough that DeID-GPT's category coverage informs us.
