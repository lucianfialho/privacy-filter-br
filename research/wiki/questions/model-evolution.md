## v3 — production baseline (pre-research)

Pre-existing model. BERTimbau fine-tune on 49842 examples generated by `{4devs perfis + Jinja templates + Claude Haiku CLI rewrite + exact-string labeler}`. Holdout F1 0.9900 micro / 0.9934 macro.

**The shape of v3 we inherited:** 22 categories, 374877 entity spans, generated against Haiku-style prose. The model itself was reasonable; the data pipeline had a silent bug that nobody had measured yet.

## v3.1 — first iteration (filter+retrain, marginal gains)

**Hypothesis:** if regex-augmented relabeling catches PII the original labeler missed, F1 should rise.

**What we found in the audit before training:**

- **16.76% of v3 examples (8355/49842) had at least one regex-detectable PII pattern unlabeled.** Per-pattern miss-rate: email 6.43%, CPF 6.72%, phone 4.63%, titulo_eleitor 7.79%, RG 7.36%.
- **22 categories, 374877 spans** in v3. Heavy skew: private_person 94%, private_email 80%, private_cpf 80%; under-represented: account_number, url, secret, **date 3.64%** (instrumentation bug — labeler never inserts date as PII slot).
- **v3 disagrees with its own gold on 99.30% of examples** (sample 1000). v3 finds MORE entities than labeled almost everywhere. Confidence is uniformly high (mean 0.995) — so confidence thresholds won't filter the noise.
- **Smoking-gun example:** line 0 has `**CPF:** 320.575.016.04`. Inserted CPF was `320.575.016-04`; rewriter swapped `-` for `.`. Exact-string labeler dropped it.

Relabel via `scripts/relabel_dataset.py` added **6209 new labels across 5503 examples (11.04%)**. Big winners: email (+2944), CPF (+1546), phone (+1003). Skipped categories where regex would create false positives (cep, cnh, account_number, etc).

**Training:** BERTimbau fine-tune on relabeled data, same hyperparams as v3 (3 epochs, batch 8×2, lr 3e-5).

**Result on v3's original holdout (BIASED — same labeler bug):**

| | precision | recall | F1 | Δ vs v3 |
| --- | --- | --- | --- | --- |
| v3 micro | 0.9859 | 0.9942 | 0.9900 | baseline |
| v3.1 micro | 0.9724 | 0.9953 | 0.9837 | **-0.63** |

CPF recall jumped from 0.9935 → **1.0000** (v3.1 catches every format variant). Precision dropped on every relabeled category because the holdout still didn't have the variants labeled.

**2×2 holdout benchmark (after relabeling holdout with same pipeline):**

|  | holdout original | holdout relabeled |
| --- | --- | --- |
| v3 | 0.4430 | 0.4406 ↓ |
| v3.1 | 0.4406 | **0.4461** ↑ |

(Set-match scoring here, not seqeval — absolute numbers are lower but the crossover is the signal.) When the gold is fair, v3.1 wins. When the gold has the bug, v3 wins. **Holdout-bug hypothesis confirmed.**

**Lesson:** relabel-after-the-fact can only recover regex-detectable variants. Free-text categories (names, addresses, IDs varied beyond format) need root-cause fix at generation time. Hence Phase 2.

## Phase 2 — format-aware labeler (root-cause fix)

**The fix:** `src/labeler.py` rewritten.

1. **Always exact match** (`re.escape(value)`) preserved.
2. **Plus format-aware match for structured labels** (CPF / CNPJ / RG / PIS / CNH / titulo_eleitor / IE / phone / certidao): extract alnum skeleton (`"123.456.789-09"` → `"12345678909"`), build regex allowing ≤2 separator chars between consecutive alnum positions, anchor at word boundaries.
3. **Free-text labels** stay exact match.
4. **Dedup by longest span** preserved.

Added 10 labeler tests; all 38 project tests pass.

**Validation on regenerated 5448-example smoke dataset:**

- Examples with unlabeled regex PII: **16.76% (v3) → 5.21% (v4-smoke).** Remaining 5.21% is the `financeiro@empresa.com.br` template placeholder (template bug, not labeler).
- Per-category miss rates: CPF 6.72% → **0.00%**, CNPJ 0.32% → **0.00%**, RG 7.36% → **0.00%**, titulo_eleitor 7.79% → **0.00%**, phone 4.63% → **0.17%**.
- Format diversity now present: CPF has canonical / masked / raw / spaces variants all labeled, ~1k each. In v3 only canonical got labels.

**Smoke training (5448 examples, 13 min RTX 2070 SUPER):**

| | v3 final | v3.1 ep3 | v4-smoke ep3 |
| --- | --- | --- | --- |
| F1 micro | 0.9900 | 0.9837 | **0.9983** |
| training set | 50000 | 50000 | **5448** (10× smaller) |

**v4-smoke with 10× less data beats v3.** Label quality > volume — at least within a single rewriter style.

Critical limitation (only realized later): the smoke test used the same gpt-5-nano rewriter as the full v4 dataset. The label-quality gain was real, but **stylistic monoculture was untested**.

## v4 — full scale, then catastrophic cross-holdout failure

**Pipeline:** 50000 examples + 5000 extras via OpenAI Batch API + gpt-5-nano (reasoning_effort=minimal). Format-aware labeler applied. 99.74% valid main + 99.9% valid extras. Combined: 49900 train + 4964 holdout.

**Training:** 3 epochs, 1h55min RTX 2070 SUPER.

**Result on v4's own holdout:**

| | v3 final | v4 final |
| --- | --- | --- |
| eval_loss | — | **0.0024** (20× smaller) |
| F1 micro | 0.9900 | **0.99923** (+0.0092) |
| F1 macro | 0.9934 | **0.9989** |

**13 of 22 categories at perfect F1 1.0000.** Biggest win: private_phone 0.9627 → 0.9989. Only regression: private_rg -0.0080. Looked like a clean victory.

**The user paused before PyPI publish and asked "is this really better than v3?"** That instinct saved the project. Cross-holdout benchmark:

|  | holdout_v3 (Haiku-generated) | holdout_v4 (gpt5nano-generated) |
| --- | --- | --- |
| v3 | 0.9901 ✓ | 0.9947 |
| v4 | **0.5534** ⚠️ | 0.9992 ✓ |

v4 failed on **all 22 categories** with similar TP/FP/FN ratios when run against v3's holdout. Same-position relabel? Only 3 examples out of 4929 had v4 picking a different category. The issue was **boundary bleed into markdown** — v4 learned that PII is surrounded by gpt5nano's `- **CPF:** ...` structure and extended predictions into the markup.

Full diagnostics in [[2026-05-24-v4-postmortem]] (kept as standalone — it's a complete incident report with root cause, scripts, and lessons).

**Actions taken:**

- Reverted Gradio Space + lib docs to v3.
- Did NOT publish `br-pii-guardrail` 0.1.2 to PyPI.
- Kept v4 on HF Hub as research artifact (don't break existing download URLs).

## v5 — mixed rewriters

**Hypothesis (from v4 post-mortem):** train on the union of Haiku-rewritten (v3) and gpt5nano-rewritten (v4) datasets → model can't overfit to either monoculture.

**Setup:**

- `dataset_br_v5.jsonl` = concat of v3 dataset (49842, Haiku) + v4 dataset (49900, gpt5nano) = **99742 examples**
- Holdout: combined v3+v4 holdouts = 9893 examples
- Same hyperparams; 3h45min RTX 2070 SUPER. **Cost: $0** (reused datasets).

**Result:**

| Model | Haiku-style holdout | gpt5nano-style holdout | Spread |
| --- | --- | --- | --- |
| v3 | 0.9901 | 0.9947 | 0.0046 |
| v4 | 0.5534 ⚠️ | 0.9992 | 0.4458 💥 |
| **v5** | **0.9912** ✓ | **0.9992** ✓ | **0.0080** ✓ |

55× smaller spread than v4. v5 strictly dominates v3 (+0.0011 / +0.0045) and v4 on Haiku (+0.4378).

**Published:** v5 to HF Hub, Space updated, `br-pii-guardrail` bumped 0.1.2 → 0.1.3.

**v5's known limit (which became v6's brief):** the test input `"Pedido ML-2024-789456 do cliente CUST-998877. ... faturou R$ 50.000,00 ..."` failed in two specific ways:

1. CUST-998877 → labeled as `private_order_id` (should be `private_customer_id`)
2. R$ 50.000,00 → not detected at all (should be `private_client_revenue`)

These are **template coverage gaps**, not rewriter or data-quality issues. Both ML-X and CUST-X live in templates that put them in "Pedido X / Cliente X" contexts; the model has no semantic signal to disambiguate. Revenue almost never appears in free-form prose in v5 templates (it's in tables/dashboards).

## v6 — narrative templates

**Hypothesis:** add 10 narrative templates with explicit semantic disambiguation ("o cliente CUST-X", "pedido ML-X") and revenue in prose ("faturou R$ X em outubro"). Mix into v5 dataset → fix template coverage without regressing structured-doc performance.

**The 10 templates:** artigo_noticia, email_conversacional, doc_tecnico, nota_livre, dialogo_chat, email_thread, comentario_sistema, artigo_blog, rh_perfil_narrativo, incident_report.

**Setup:**

- 30000 new examples via gpt-5-nano batch (~$1.50, 30 min)
- 29771 valid; combined with v5 = **126757 train + 12649 holdout**
- Volume jump on disambiguation-relevant cats: customer_id 7.5k → 33k, client_revenue 16k → 46k, order_id 10k → 37k
- Same hyperparams; ~5h training

**Result on the 3-style cross-benchmark:**

| Model | Haiku struct | gpt5nano struct | gpt5nano narrative | Spread |
| --- | --- | --- | --- | --- |
| v5 | 0.9912 | 0.9992 | 0.9564 | 0.0429 |
| **v6** | **0.9909** | **0.9993** | **0.9991** | **0.0084** |

**+0.0427 on narrative, ≈ tied on the other two.** Spread shrank 5×. The two concrete failing inputs from v5 (CUST-998877 → customer_id 0.9999, R$ 50.000,00 → client_revenue 0.9980) both fixed.

**Published:** v6 to HF Hub, Space, lib 0.1.3 → 0.1.4.

**v6's known limit (which became v7's brief):** Phase 1 real-world benchmark on 30 CVM FRE 2024 board-member docs revealed:

- **Synthetic holdout: F1 0.997.**
- **Real CVM data, exact match: F1 0.0000** (yes, zero).
- **Real CVM data, overlap-tolerant: F1 cpf 1.0000, cnpj 0.97, person 0.16, date 0.00, macro ~0.53.**

Three concrete root causes diagnosed:

1. **Universal boundary fragmentation** — sub-token predictions never merged back into spans. Exact-match scoring catches none.
2. **ALL-CAPS names absent from training** — CVM docs use `JOÃO MARIA DA SILVA`; training had only Title Case.
3. **Dates in narrative context absent** — "nascido em 12/03/1978" never appeared in training prose.

## v7 — Phase 1 fixes (in flight)

Three targeted fixes, each addressing one diagnosed cause:

**Fix 1 — boundary merger as runtime post-process** in `br-pii-guardrail/src/br_pii_guardrail/ner.py`. `_merge_adjacent_spans()` merges contiguous predictions of same label separated by ≤3 chars of separator content (`.-/() \t  \n`). 8 unit tests in `br-pii-guardrail/tests/test_merge_spans.py` covering CPF/CNPJ/multi-word person fragments, different-label non-merge, far-apart non-merge, alphanumeric-gap non-merge, three-fragment chain merge, score = min of fragments, empty input. All passing.

**Fix 2 — ALL-CAPS name variants** in `scripts/openai_batch.py`. `inserted` dict now registers `nome`, `nome.upper()`, `nome.title()`; `chosen` exposes `nome_upper` for templates.

**Fix 3 — three new templates with formal CAPS + date-in-prose:**

- `templates/ata_assembleia.jinja2` — ATA DA ASSEMBLEIA GERAL with `{{ nome_upper }}`, `nascido em {{ data_nasc }}`
- `templates/edital_oficial.jinja2` — EDITAL DE INTIMAÇÃO with `{{ nome_upper }}` in CAPS opening
- `templates/cadastro_administrador.jinja2` — mixed CAPS header + Title Case body, dates in prose

Added to TEMPLATES list in `scripts/openai_batch.py` (now 31 total).

**Batch:** 15000 examples via gpt-5-nano. **14902 valid (99.35%).** Combined with v6: **140297 train + 14011 holdout.**

**Training:** 5h51min RTX 2070 SUPER. **Synth F1:** 0.9968 micro.

**Phase 1 CVM result:** cpf 1.00 / cnpj 1.00 / person 0.71 / **date 0.00** / macro 0.68 (overlap).

Boundary merger transformed exact F1 from 0.00 to ~0.90 — confirmed the universal fragmentation hypothesis. CAPS variants helped person detection. **But date stayed at zero.** Lib bumped to 0.1.5, model published as v7 tag but not promoted to main (waiting on date fix).

## v8 — instrumentation fix (data_nasc registered)

**Root cause for v7 date failure:** `data_nasc` and `endereco` were rendered into Jinja templates via `chosen` dict but never registered in `inserted` dict that the labeler uses. Result: across all 140k training examples, **zero** `private_date` labels existed for the prose contexts. Model couldn't learn what it never saw labeled.

**Fix:** in `src/batch/instrumentation.py` (post-refactor of openai_batch.py — issue #4):

```python
# Date variants
inserted[data_nasc] = "PRIVATE_DATE"
inserted[data_nasc.replace("/", "-")] = "PRIVATE_DATE"
inserted[data_nasc.replace("/", ".")] = "PRIVATE_DATE"

# Full street address + partial
inserted[endereco] = "PRIVATE_ADDRESS"
parts = endereco.split(",")
if len(parts) >= 2:
    inserted[",".join(parts[:2]).strip()] = "PRIVATE_ADDRESS"

# Name variants — also added Title Case + lowercase + accent-stripped
nome_ascii = _strip_accents(nome)
for variant in {nome, nome.upper(), nome.title(), nome.lower(),
                nome_ascii, nome_ascii.upper(), nome_ascii.title(), nome_ascii.lower()}:
    inserted[variant] = "PRIVATE_PERSON"
```

**Dataset:** v7 + 30k new gpt-5-nano + 5k extras = 172009 train / 17070 holdout. **Audit confirmed:** `private_date` jumped from 0 to 35804 labels (20.8% docs); `private_address` from CEP-only to full street + partial.

**Training:** 7h11min. **Synth F1:** 0.9898 micro, date F1 0.95 synth.

**Phase 1 CVM:** cpf 1.00 / cnpj 1.00 / person 0.73 / **date 0.00** ❌ / macro 0.71. Date still failed.

**Diagnosis:** model learned date IN gpt-5-nano's specific phrasing (`"dia X foi"`, `"nascido(a) em X, filho(a)"`) but CVM writes `"nascido em X, exerce profissão"`. Different connector around the date token → model goes from 99% confident to 2% confident. Same class of bug as v4 but specific to one category.

## v8.1 — public-doc templates fix date context overfit

**Hypothesis:** the date overfit is not a labeling problem (v8 fixed that) but a phrasing-context problem. Force the rewriter to preserve real BR phrasings by giving it templates that mirror real documents.

**Seven new templates** added in `templates/`:

- `cvm_fre_administrador` — CVM FRE board-member format
- `rfb_cnpj_cadastro` — Receita Federal CNPJ lookup
- `jucesp_ato_societario` — JUCESP partnership alteration
- `dou_publicacao_oficial` — DOU edital format
- `sintegra_consulta` — SINTEGRA contributor lookup
- `b3_fato_relevante` — B3 material fact disclosure
- `timeline_evento` — date-rich timeline with 10+ phrasings (`nascido em`, `nato em`, `vigência X a Y`, `desde Z`, `aniversário`, `em DD/MM/YYYY`)

**Dataset:** v7 + 30k new gpt-5-nano (with new templates) + 5k extras = 172075 train / 17072 holdout. Audit: `private_date` 37.7% docs, `private_address` 81.4%.

**Training:** 6h57min, seed=42 pinned. **Synth F1:** 0.9906 micro, per-category: date F1 0.9616 (precision 0.96, recall 0.96), address F1 0.9375.

**Phase 1 CVM (overlap, with merger):**

| | cpf | cnpj | person | **date** | macro4 |
| --- | --- | --- | --- | --- | --- |
| v8.1 | 1.0000 | 1.0000 | 0.7143 | **0.7500** 🎯 | **0.8536** |

Was 0.0000 in v7 and v8. **TP 24, FP 10, FN 6 on 30 documents.** Exact F1 with merger: 0.3125 (TP 10 boundary-perfect, FP 24 boundary-off). Model now detects dates in real CVM; some boundaries need polish but the existence-detection problem is solved.

**Published v8.1 to HF canonical** (main + tag `v8.1`). Lib bumped 0.1.5 → 0.1.6.

**Lesson reinforced:** template diversity matters more than dataset size for closing context-overfit gaps. The new templates added ~10% to the dataset but moved date F1 from 0 to 0.75 in real-world. Same class of fix as v6 narrative templates (which fixed CUST/revenue gaps) — context shape is what the model overfits on, and the cheapest way to teach robustness is to give the rewriter a more constrained template that copies real-world phrasing patterns it'll have to preserve.

## Cumulative lessons (across all versions)

1. **A 99% F1 on your own holdout means "consistent with my training," not "good model."** Cross-holdout is the cheap honesty check. Synth-vs-real is the expensive but mandatory one.
2. **Single-source LM rewriters create stylistic monoculture.** LAMBADA warned about this; v4 demonstrated it empirically.
3. **Label quality > volume — but ONLY within a fixed style.** v4-smoke proved 10× less data wins on the same distribution. Same data on a different distribution loses 44 F1 points.
4. **Boundary fragmentation hides in seqeval BIOES F1 on synthetic holdouts** because gold and predictions are produced by the same tokenizer + tagger. Real-world data breaks this invariant; runtime span merging fixes it cheaply.
5. **Template coverage > rewriter diversity for downstream "shape" matters.** v6 added templates only (no new rewriter) and gained +0.0427 on narrative. Diversity of CONTENT shape, not just producers.
6. **Pause before publishing.** The instinct "is this really better?" is the difference between shipping a regression and not.
7. **Synthetic-to-real gap is its own axis.** Cross-holdout fixes synthetic-to-synthetic. The real gap shows up only against real documents.
